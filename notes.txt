"""
Option 1: Move Logic to Separate Files (Recommended)
This approach promotes a clean separation of concerns and is generally the best practice for maintainable and scalable applications, especially when integrating multiple frameworks like FastAPI and Django.
Proposed Structure:
*   fastapi_app/api.py:
    *   This file would contain the core FastAPI application instance (app = FastAPI(...)) and all endpoint definitions (@app.get, @app.post).
    *   These endpoints would primarily act as orchestrators, calling methods from various service classes.
*   fastapi_app/dependencies.py:
    *   Move get_llm() and any other FastAPI-specific dependency injection functions here.
*   fastapi_app/services/ (New Directory): This directory would house all business logic.
    *   fastapi_app/services/task_manager.py: Encapsulates the high-level task coordination logic.
        *   Contains perform_scraping_task and perform_matching_task.
        *   Responsible for updating task statuses in the database.
        *   Coordinates calls to ScraperService, OllamaService, and DjangoDBService.
    *   fastapi_app/services/scraper_service.py: Handles all scraping-related operations.
        *   Contains _get_scraper_for_portal, _process_job_listing, and _scrape_portal.
        *   The scraper classes (JustJoinITScraper, PracujplScraper, TheProtocolITScraper) could either be defined here or in a fastapi_app/scrapers/ directory if they become complex.
    *   fastapi_app/services/ollama_service.py: (As discussed previously) This class manages the openai.Client instance and the make_llm_call logic.
    *   fastapi_app/services/django_db_service.py: This service would abstract direct Django ORM calls.
        *   Contains _get_task_record (which would be renamed to a public method).
        *   Handles JobListing.objects.update_or_create, Task.objects.create, Resume.objects.get, etc.
        *   This layer is critical for future decoupling if you decide to interact with Django via its own API instead of direct model imports.
*   fastapi_app/core/django_setup.py:
    *   Contains the Django environment setup (os.environ.setdefault, django.setup()) and Django model imports.
    *   api.py (or a central main.py if it becomes very thin) would call a setup function from this module.
*   fastapi_app/config.py:
    *   For shared configurations, including logging setup.
Pros of Option 1:
*   Clear Separation of Concerns: Each module has a single, well-defined responsibility, making the codebase easier to understand and navigate.
*   Improved Readability and Maintainability: Smaller, focused files reduce cognitive load and make changes less prone to side effects.
*   Enhanced Testability: Services can be easily unit-tested by mocking their dependencies.
*   Scalability: Easier to add new features or modify existing ones without creating a monolithic file.
*   Facilitates Decoupling: Paves the way for replacing direct Django ORM calls with a Django API client by centralizing DB interactions in DjangoDBService.
Cons of Option 1:
*   Requires creating several new files and directories, increasing initial refactoring effort.
"""
